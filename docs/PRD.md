## 2. 제품 요약
Vibe Studio는 AI 입문자가 막연한 생각, 초안, 요청을 더 선명한 형태로 바꾸도록 돕는 학습형 작업대다.

이 제품은 단순히 결과물만 생성하는 도구가 아니다.
사용자가 자신의 의도를 구조화하고, 필요한 정보를 보완하고, 더 적절한 결과물 방향을 선택하도록 돕는 것이 핵심이다.

MVP는 아래 네 가지 결과 방향을 지원한다.
- 프롬프트 (`prompt`)
- 기획형 구조 정리 (`spec`)
- 구조 설계 (`architecture`)
- 검토 리포트 (`review-report`)

## 3. 타깃 사용자와 문제 정의
핵심 타깃은 AI 입문자다.

이들은 보통 아래 문제를 겪는다.
- 무엇을 요청해야 할지 잘 모른다
- 머릿속 아이디어를 구조적으로 표현하기 어렵다
- 프롬프트, 기획, 구조 설계, 검토의 차이를 잘 모른다
- 결과가 아쉬워도 무엇이 빠졌는지 설명하기 어렵다

Vibe Studio는 이런 사용자가:
- 더 구체적으로 말하고
- 더 적절한 결과 방향을 이해하고
- AI와 협업하는 감각을 익히도록 돕는 것을 목표로 한다

## 4. MVP 목표
MVP의 목표는 아래와 같다.
- 사용자가 자유 입력으로 시작할 수 있게 한다
- 시스템이 생성 요청과 검토 요청을 구분할 수 있게 한다
- 시스템이 현재 요청에 맞는 결과 방향을 추천할 수 있게 한다
- 핵심 정보가 부족하면 먼저 질문할 수 있게 한다
- 필요한 경우 바로 생성하지 않고 확인 단계를 거치게 한다
- 초보자가 왜 질문을 받는지 이해할 수 있는 경험을 제공한다

## 5. MVP 비목표
이번 버전에서 우선하지 않는 것은 아래와 같다.
- 숙련 사용자별 적응형 정책
- 사용자 수준별 다른 승인 동작
- 계정 기반 개인화
- 협업 기능
- 복잡한 저장/버전 관리
- 세밀한 모델 설정 UI
- 전문가용 파워툴 수준의 고급 제어
- 고급 편집기 기능

Rule:
- MVP는 예측 가능성과 학습 경험을 우선한다
- 개인화와 고급 최적화는 후속 단계로 미룬다

## 6. 핵심 사용자 흐름
### Flow A. 생성 흐름
1. 사용자가 자유롭게 입력한다
2. 필요하면 카드를 보조적으로 선택한다
3. 시스템이 입력의 의도를 해석한다
4. 시스템이 적절한 결과 방향을 추천한다
5. 핵심 정보가 부족하면 먼저 질문한다
6. 정보가 충분하면 필요에 따라 확인 단계를 거친다
7. 최종 결과를 생성한다

### Flow B. 검토 흐름
1. 사용자가 기존 초안이나 결과물을 가져온다
2. 시스템이 검토 요청으로 해석한다
3. 검토 대상이 없으면 먼저 보완을 요청한다
4. 검토 대상이 있으면 검토 리포트를 생성한다

### Flow C. 카드 불일치 흐름
1. 사용자가 카드를 선택하고 입력한다
2. 시스템이 선택 카드와 실제 의도를 함께 본다
3. 강한 불일치가 있으면 더 적절한 방향 전환을 제안한다
4. 시스템은 사용자 승인 없이 조용히 방향을 바꾸지 않는다

## 7. 핵심 제품 요구사항
### 7.1 입력 시작
- 사용자는 카드 선택 없이도 바로 입력할 수 있어야 한다
- 카드는 필수 단계가 아니라 보조 진입 장치여야 한다

### 7.2 의도 해석
- 시스템은 최소한 생성 요청과 검토 요청을 구분할 수 있어야 한다
- 시스템은 현재 요청에 맞는 결과 방향을 추천할 수 있어야 한다

### 7.3 질문과 보완
- 핵심 정보가 없으면 결과 생성 전에 질문을 던져야 한다
- 질문은 많기보다 중요한 정보 위주여야 한다
- 사용자는 왜 이 질문을 받는지 이해할 수 있어야 한다

### 7.4 확인 단계
- 시스템은 아래 세 상태를 구분할 수 있어야 한다
  - 바로 생성 가능
  - 먼저 질문 필요
  - 확인 후 진행 필요
- 확인 단계는 권장 확인과 필수 확인을 구분해야 한다

### 7.5 결과 생성
- 시스템은 아래 네 가지 결과 방향을 지원해야 한다
  - 프롬프트 (`prompt`)
  - 기획형 구조 정리 (`spec`)
  - 구조 설계 (`architecture`)
  - 검토 리포트 (`review-report`)

### 7.6 방향 전환 제안
- 사용자가 고른 카드와 실제 과업 성격이 강하게 다르면 방향 전환을 제안해야 한다
- 시스템은 사용자 승인 없이 카드나 결과 방향을 조용히 바꾸면 안 된다

## 8. 결과 방향 범위
MVP에서 다루는 결과 방향은 아래와 같다.

### 프롬프트 (`prompt`)
사용자가 바로 활용할 수 있는 실행형 결과

### 기획형 구조 정리 (`spec`)
아이디어를 문제, 대상, 목표, 방향이 보이는 구조로 정리한 결과

### 구조 설계 (`architecture`)
서비스, 시스템, 흐름, 구성요소 관계를 잡아주는 결과

### 검토 리포트 (`review-report`)
기존 초안, 프롬프트, 기획안, 구조에 대한 문제점과 보완 포인트를 정리한 결과

## 9. UX 요구사항
- 첫 화면은 입력 중심이어야 한다
- 카드는 선택 보조장치여야 한다
- 시스템은 먼저 이해한 바를 비춰줘야 한다
- 질문은 왜 필요한지 이해 가능한 방식으로 제시되어야 한다
- 마찰은 처벌처럼 느껴지지 않아야 한다
- 초보자가 기술 용어를 몰라도 흐름을 따라갈 수 있어야 한다
- 시스템은 블랙박스처럼 보이지 않아야 한다
- 생성과 검토의 차이가 사용자 경험에서 자연스럽게 드러나야 한다

## 10. 성공 기준
MVP는 아래가 가능하면 성공으로 본다.
- 초보자 입력에서 적절한 결과 방향 추천이 가능하다
- 핵심 정보 부족 시 먼저 질문하는 흐름이 자연스럽게 작동한다
- 생성과 검토가 일관되게 구분된다
- 권장 확인과 필수 확인이 서로 다른 경험으로 구분된다
- 카드 오선택 시 방향 전환 제안이 일관되게 동작한다
- 사용자가 왜 질문을 받는지 이해할 수 있다

## 11. 후속 범위
후속 단계에서 검토할 범위는 아래와 같다.
- 숙련 사용자에 대한 적응형 정책
- 승인 규칙의 세밀한 튜닝
- 더 많은 대표 사례 추가
- 저장과 히스토리 기능
- 협업 기능
- 고급 사용자용 세부 제어
