# Product-Web Manual Checklist

Updated: 2026-05-15
Purpose: `apps/product-web`를 기준으로 실제 입문자 관점의 단순 입력, 바로 결과, 후속 행동 흐름을 점검한다.

## Rule
- 이 문서는 `product-web` 수동 점검 기준 문서다
- 현재 브라우저 수동 검증의 기본 기준면으로 쓴다
- source-of-truth 제품 문서를 덮어쓰지 않는다
- 입문자 기본 화면은 질문 카드와 승인 카드 없이 시작 방식 카드에서 템플릿 입력 또는 자율입력으로 이어지는 방향을 따른다

Note:
- `apps/prompt-web`는 동결된 최소 fallback 검증 서페이스로 남아 있을 수 있다
- 하지만 제품 수동 점검의 기본 기준면은 계속 `product-web`다

## 1. 왜 필요한가
실제 입문자 피드백에서 별도 질문과 버튼이 오히려 사용자를 멈추게 할 수 있음이 확인됐다.

따라서 `product-web`의 기본 흐름은 아래처럼 본다.

1. 사용자가 시작 방식 카드를 고른다
2. 카드에 맞는 짧은 템플릿을 채우거나 자율입력을 적는다
3. 시스템이 먼저 결과를 만든다
4. 부족한 정보와 주의점은 결과 안에서 설명한다
5. 필요한 후속 action은 결과 아래에만 붙인다

## 2. 테스트 준비
수동 점검 전 확인:

1. `product-server`가 실행 중이어야 한다
2. `product-web`이 실행 중이어야 한다
3. 로컬 provider 또는 유효한 remote provider 세션이 준비되어야 한다
4. 브라우저는 새 세션 상태에서 시작하는 편이 좋다

## 3. 합격 질문
각 시나리오는 아래 질문에 답할 수 있어야 한다.

- 첫 화면에서 사용자가 눌러야 할 선택지가 과하게 많지 않은가
- 빈 입력창 앞에서 막막하지 않게 시작 방식 카드가 먼저 보이는가
- 짧고 어설픈 입력도 먼저 결과로 이어지는가
- 부족한 정보가 질문 카드가 아니라 결과 안의 열린 지점으로 보이는가
- 결과 본문 전에 내가 이해한 요청, 추천 작업 형태, 빠진 정보, 조심할 추측이 먼저 보이는가
- 결과 초안은 API 연결 없이도 먼저 볼 수 있는 정리본처럼 보이는가
- 결과를 본 뒤 다음에 더 잘 요청하려면 무엇을 덧붙이면 되는지 말할 수 있는가
- review와 create가 실제로 다른 결과처럼 느껴지는가
- 원본 결과와 후속 결과가 섞이지 않는가

## 4. 시나리오
### A. 첫 화면 단순성
확인 포인트:
- 시작 방식 카드가 첫 화면의 중심이어야 한다
- 예시 버튼 묶음이 보이지 않아야 한다
- 결과 방향 카드나 hint chip이 보이지 않아야 한다
- pre-result 질문 카드나 approval 카드가 보이지 않아야 한다
- 자율입력은 별도 카드로만 열려야 한다

합격 조건:
- 사용자가 첫 화면에서 `자율입력`, `프롬프트`, `플랜`, `아키텍처`, `검증 및 리뷰` 중 하나를 고를 수 있다

### B. 플랜 템플릿 Direct Result
입력 예시:
카드: `플랜`
만들고 싶은 것: `개인적인 헤지펀드 매니저`

확인 포인트:
- 카드 선택 없이 바로 입력 가능해야 한다
- 결과가 `plan` 성격으로 나온다
- 중간 질문이나 승인 화면이 끼어들지 않는다
- 결과 화면이 `먼저 이렇게 이해했어요` 다음 `결과 초안` 순서로 읽힌다
- 사용자가 결과를 보고 부족한 대상/범위 정보를 자기 말로 설명할 수 있다
- AI 코딩툴 복사용 내용은 plan 결과보다 먼저 나오지 않고, 결과 이후 보조 next step으로 보여야 한다

합격 조건:
- 결과가 바로 나온다
- 부족한 대상/범위 정보가 결과 안에 열린 지점으로 남는다

### C. 부족한 Architecture 입력
입력 예시:
`내가 만들고 싶은 앱이 있는데 사용자용 화면, 가게용 화면, 결제, 알림이 다 들어가. 전체 구조를 먼저 잡아줘.`

확인 포인트:
- 예전처럼 질문 카드에서 멈추지 않아야 한다
- 가능한 범위의 구조 결과가 먼저 나와야 한다
- 부족한 범위, 가정, 열린 지점은 결과 안에 남아야 한다
- 결과 상단에서 왜 prompt나 plan보다 architecture에 가까운지 설명되어야 한다
- 사용자가 결과를 보고 "이번에는 무엇을 가정하고 구조를 잡았는지" 말할 수 있어야 한다

합격 조건:
- 결과가 바로 나온다
- `구조 설계` 성격이 유지된다
- 판단 근거나 notes에서 부족한 정보가 설명된다

### D. Recommended Approval Plan 입력
입력 예시:
`동네 커피 모임 앱 아이디어를 MVP 기획 정리로 잡아줘.`

확인 포인트:
- approval 화면에서 멈추지 않아야 한다
- 기획 정리 결과가 먼저 나와야 한다
- 후속 action은 결과 아래의 `한 번 더 살펴보기` 보조 행동으로만 보여야 한다

합격 조건:
- 결과가 바로 나온다
- 후속 action이 원본 결과 아래에 붙는다

### E. Required Approval Architecture 입력
입력 예시:
`주문 관리 앱 전체 구조를 잡아줘. 범위는 사용자 앱, 점주 앱, 관리자 페이지, 결제 서비스, 알림 서비스고 주문 생성, 결제 승인, 주문 상태 알림 흐름에 집중해줘.`

확인 포인트:
- required 신호가 있어도 pre-result 승인 버튼으로 멈추지 않는다
- 구조 설계 결과가 먼저 나온다
- 주의할 범위와 확인할 가정이 결과 안에서 읽혀야 한다
- `API 연결`이나 `모델 설정`이 기본 진행 조건처럼 보이지 않아야 한다

합격 조건:
- 결과가 바로 나온다
- 결과가 구현 지시나 코드 생성으로 점프하지 않는다

### F. Review 흐름
입력 예시:
`이 소개 문구 초안을 검토해줘: 우리 앱은 누구나 생산성을 10배 높여줍니다.`

확인 포인트:
- review 결과는 create 결과처럼 새 산출물 본문만 내지 않아야 한다
- `판단`, `finding`, `recommendation` 구조가 드러나야 한다
- review tone이 유지되어야 한다

합격 조건:
- `review-report` 결과가 나온다
- 최소 1개 이상의 actionable finding이 보인다

### G. Review Follow-Up
진입 방법:
- review 결과 아래 `지적 반영해서 다시 쓰기` 클릭

확인 포인트:
- 원본 review 결과가 사라지면 안 된다
- 후속 결과가 별도 블록으로 나와야 한다
- `무엇이 달라졌는지`가 보여야 한다
- 후속 action이 반복 수정 루프나 자유 입력 지시처럼 보이지 않아야 한다

합격 조건:
- 원본 review 블록과 후속 결과 블록이 구분된다
- 수정안이 별도 블록으로 나온다
- change summary가 보인다

### H. Plan Follow-Up
입력 예시:
`동네 커피 모임 앱 아이디어를 MVP 기획 정리로 잡아줘.`

진입 방법:
- 결과 아래 `더 구체화하기` 클릭

확인 포인트:
- 후속 결과가 `plan` 계열 확장처럼 보여야 한다
- 갑자기 `architecture`나 구현 명세처럼 점프하면 안 된다

합격 조건:
- 후속 결과 제목과 본문이 plan 상세화 성격을 유지한다

### I. Architecture Follow-Up
입력 예시:
`사장님과 손님이 함께 쓰는 예약 서비스의 MVP 구조를 잡아줘.`

진입 방법:
- 결과 아래 `세부 설계로 확장하기` 클릭

확인 포인트:
- 흐름을 더 자세히 보는 쪽으로 세부화되어야 한다
- API 명세나 코드 생성처럼 점프하면 안 된다
- out-of-scope가 보여야 한다

합격 조건:
- 흐름 단계가 더 잘게 풀린다
- `구현 작업`이나 `코드 생성`으로 이동하지 않는다

### J. Prompt Help Learning Panel
입력 예시:
`신입 PM이 기능 요청을 받았을 때 바로 쓸 질문 프롬프트를 만들어줘.`

확인 포인트:
- 결과 프롬프트가 먼저 보여야 한다
- 그 아래에 학습 패널이 붙어야 한다
- 한글 이름이 먼저 나오고 영문 용어는 괄호로 보여야 한다
- 입력 개선 힌트는 다음 자유 입력에 응용 가능한 한 줄 예시처럼 보여야 한다

합격 조건:
- 프롬프트 본문이 먼저 나온다
- `출력 형식 고정하기`
- `예시 없이 요청하기 (Zero-shot)`
- `예시를 보여주며 요청하기 (Few-shot)`
  가 보인다

## 5. 회귀 체크 최소 세트
빠르게 확인해야 할 최소 세트는 아래 항목이다.

1. 첫 화면에 예시 버튼/질문 카드/승인 카드가 없는지
2. 플랜 템플릿 입력 direct result
3. 부족한 architecture 입력도 바로 결과가 나오는지
4. recommended/required approval 입력도 pre-result 버튼 없이 결과가 나오는지
5. review 결과 + revise-from-review
6. 결과 상단 공통 context layer가 renderer 본문보다 먼저 보이는지
7. 결과 화면이 `요청 이해 -> 추천 작업 형태 -> 결과 초안 -> 정리 기준/확인할 점 -> 다음 입력 힌트 -> 한 번 더 살펴보기` 순서로 읽히는지
8. 화면 어디에서도 `Stage 1`, `renderer`, `flow-detail`, `approval` 같은 내부 용어가 사용자 안내처럼 보이지 않는지

## 6. 지금 문서의 역할
이 문서는 제품 프론트가 실제 입문자에게 질문과 버튼 부담을 줄이고, 먼저 정리된 결과를 보여주는지 눈으로 확인하는 체크리스트다.

따라서 이 문서는:
- 엔진 검증을 대체하지 않고
- `product-web` 수동 확인의 기준만 따로 고정한다
