# Product-Web Manual Checklist

Updated: 2026-04-20
Purpose: `apps/product-web`를 기준으로 초보 사용자 관점의 수동 검증 시나리오와 합격 조건을 고정한다.

## Rule
- 이 문서는 `product-web` 수동 점검 기준 문서다
- 현재 브라우저 수동 검증의 기본 기준면으로 쓴다
- source-of-truth 제품 문서를 덮어쓰지 않는다

Note:
- `apps/prompt-web`는 동결된 최소 fallback 검증 서페이스로 남아 있을 수 있다
- 하지만 제품 수동 점검의 기본 기준면은 계속 `product-web`다

## 1. 왜 필요한가
현재 Vibe Studio는 engine / renderer 검증은 강하다.
하지만 실제 사용자는 `product-web`에서:
- 입력
- 질문
- 승인
- 결과
- 후속 action
을 하나의 흐름으로 경험한다.

따라서 제품화 직전에는 `product-web` 기준 수동 점검 문서가 따로 있어야 한다.

## 2. 테스트 준비
수동 점검 전 확인:

1. `product-server`가 실행 중이어야 한다
2. `product-web`이 실행 중이어야 한다
3. 로컬 provider 또는 유효한 remote provider 세션이 준비되어야 한다
4. 브라우저는 새 세션 상태에서 시작하는 편이 좋다

## 3. 합격 질문
각 시나리오는 아래 질문에 답할 수 있어야 한다.

- 왜 이 질문이 나왔는지 사용자가 이해 가능한가
- 왜 지금 바로 생성되지 않았는지 설명 가능한가
- review와 create가 실제로 다른 경험처럼 느껴지는가
- 원본 결과와 후속 결과가 섞이지 않는가
- 사용자가 다음 행동을 의식적으로 고르게 되는가

## 4. 시나리오
### A. 자유 입력 시작
입력 예시:
`회의 전에 내가 뭘 물어봐야 할지 정리해주는 프롬프트 만들어줘`

확인 포인트:
- 카드 선택 없이 바로 입력 가능해야 한다
- 입력창이 첫 화면의 중심이어야 한다
- 보조 힌트가 입력보다 앞서 보이지 않아야 한다
- 결과가 `prompt`로 직행해야 한다

합격 조건:
- `direct_render`로 결과가 나온다
- 사용자는 카드 선택 없이 시작할 수 있다

### B. Clarify 흐름
입력 예시:
`내가 만들고 싶은 앱이 있는데 사용자용 화면, 가게용 화면, 결제, 알림이 다 들어가. 전체 구조를 먼저 잡아줘.`

확인 포인트:
- 바로 결과로 가지 않고 질문이 먼저 보여야 한다
- 질문과 함께 왜 더 알려달라는지 드러나야 한다
- 아래 입력창에서 바로 이어서 보완 가능해야 한다

합격 조건:
- stage가 `clarify`로 보인다
- 이유 문장이 함께 보인다
- 같은 입력창에서 이어서 보완할 수 있다

### C. Recommended Approval 흐름
입력 예시:
`동네 커피 모임 앱 아이디어를 MVP 기획 정리로 잡아줘.`

확인 포인트:
- `approval_pending + recommended`에 맞는 확인 화면이 보여야 한다
- 즉시 진행 버튼과 입력 보완 진입이 함께 보여야 한다
- 사용자가 "한 번 더 다듬을 수도 있고 그대로 갈 수도 있다"는 상태를 이해할 수 있어야 한다

합격 조건:
- approval 화면이 노출된다
- `입력 보완하기`가 보인다
- `그대로 진행`이 보인다

### D. Required Approval 흐름
입력 예시:
`주문 관리 앱 전체 구조를 잡아줘. 범위는 사용자 앱, 점주 앱, 관리자 페이지, 결제 서비스, 알림 서비스고 주문 생성, 결제 승인, 주문 상태 알림 흐름에 집중해줘.`

확인 포인트:
- `required` 승인 강도가 recommended와 다르게 느껴져야 한다
- 바로 넘어가기 전 범위/방향 점검 안내가 있어야 한다

합격 조건:
- approval 화면이 노출된다
- 버튼 문구가 `확인 후 진행`으로 보인다
- recommended와 copy/강도가 동일하게 느껴지지 않는다

### E. Approval Revise UX
진입 방법:
- approval 화면에서 `입력 보완하기` 클릭

확인 포인트:
- 입력 보완 화면이 단순히 사라졌다가 입력창만 남는 방식이면 안 된다
- 현재 입력이 그대로 보여야 한다
- 왜 보완이 필요한지 설명이 있어야 한다
- 보완하면 무엇이 좋아지는지 보여야 한다
- 바로 수정 입력할 수 있어야 한다

합격 조건:
- `현재 입력` 블록이 보인다
- `왜 보완이 필요한가`가 보인다
- `보완하면 무엇이 더 나아지나`가 보인다
- 아래 편집 textarea에서 바로 수정 가능하다

### F. Review 흐름
입력 예시:
`이 소개 문구 초안을 검토해줘: 우리 앱은 누구나 생산성을 10배 높여줍니다.`

확인 포인트:
- review 결과는 create 결과처럼 새 산출물 본문만 내지 않아야 한다
- `판단`, `finding`, `recommendation` 구조가 드러나야 한다
- review tone이 유지되어야 한다

합격 조건:
- `review-report` 결과가 나온다
- `판단`이 보인다
- 최소 1개 이상의 actionable finding이 보인다

### G. Review Follow-Up
진입 방법:
- review 결과 아래 `지적 반영해서 다시 쓰기` 클릭

확인 포인트:
- 원본 review 결과가 사라지면 안 된다
- 후속 결과가 별도 블록으로 나와야 한다
- `무엇이 달라졌는지`가 보여야 한다

합격 조건:
- 원본 review 블록과 후속 결과 블록이 구분된다
- 수정안이 별도 블록으로 나온다
- change summary가 보인다

### H. Review Refinement
진입 방법:
- review follow-up 아래 `남은 질문 또는 주의점` textarea에 답 입력

확인 포인트:
- 답을 입력하고 다시 보완할 수 있어야 한다
- follow-up이 같은 맥락 안에서 보강되어야 한다
- 새 renderer로 점프하거나 추가 action chain처럼 보이면 안 된다

합격 조건:
- 답변 입력창이 보인다
- `답변 반영해서 수정안 보완하기` 버튼이 보인다
- 결과가 같은 follow-up 블록 맥락 안에서 갱신된다

### I. Plan Follow-Up
입력 예시:
`동네 커피 모임 앱 아이디어를 MVP 기획 정리로 잡아줘...`

진입 방법:
- 결과 아래 `더 구체화하기` 클릭

확인 포인트:
- 후속 결과가 `plan` 계열 확장처럼 보여야 한다
- 갑자기 `architecture`나 구현 명세처럼 점프하면 안 된다

합격 조건:
- 후속 결과 제목과 본문이 plan 상세화 성격을 유지한다
- out-of-scope 없이도 여전히 기획 정리 계열로 읽힌다

### J. Architecture Follow-Up
입력 예시:
`사장님과 손님이 함께 쓰는 예약 서비스의 MVP 구조를 잡아줘...`

진입 방법:
- 결과 아래 `세부 설계로 확장하기` 클릭

확인 포인트:
- 기본 세부화 초점이 `flow-detail`이어야 한다
- API 명세나 코드 생성처럼 점프하면 안 된다
- out-of-scope가 보여야 한다

합격 조건:
- 흐름 단계가 더 잘게 풀린다
- `구현 작업`이나 `코드 생성`으로 이동하지 않는다
- out-of-scope가 보인다

### K. Prompt Help Learning Panel
입력 예시:
`신입 PM이 기능 요청을 받았을 때 바로 쓸 질문 프롬프트를 만들어줘.`

확인 포인트:
- 결과 프롬프트가 먼저 보여야 한다
- 그 아래에 학습 패널이 붙어야 한다
- 항상 보이는 포인트가 과도하게 많지 않아야 한다
- 한글 이름이 먼저 나오고 영문 용어는 괄호로 보여야 한다

합격 조건:
- 프롬프트 본문이 먼저 나온다
- `출력 형식 고정하기`
- `예시 없이 요청하기 (Zero-shot)`
- `예시를 보여주며 요청하기 (Few-shot)`
  가 보인다

### L. Pivot / Wrong Card
입력 예시:
`이 소개 문구 초안을 검토해줘...`

진입 방법:
- `프롬프트 도움` 힌트를 고른 뒤 제출

확인 포인트:
- 실제 결과는 review로 가야 한다
- 사용자가 선택한 힌트와 실제 결과가 다를 수 있다는 점이 explainable해야 한다
- 조용한 silent switch처럼 느껴지면 안 된다

합격 조건:
- 결과는 `review-report`로 나온다
- create 결과로 잘못 가지 않는다

## 5. 회귀 체크 최소 세트
빠르게 확인해야 할 최소 세트는 아래 6개다.

1. 자유 입력 prompt direct render
2. clarify-first architecture
3. recommended approval plan
4. required approval architecture
5. review 결과 + revise-from-review
6. prompt help learning panel 노출

## 6. 지금 문서의 역할
이 문서는 제품 프론트가 MVP 기준을 실제 사용자 경험으로 번역하고 있는지 눈으로 확인하는 체크리스트다.

따라서 이 문서는:
- 엔진 검증을 대체하지 않고
- `product-web` 수동 확인의 기준만 따로 고정한다
