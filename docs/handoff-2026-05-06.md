# Handoff 2026-05-06

Purpose: 다음 세션이 product-web 제품화 전 구조 정리 의도를 바로 이어받기 위한 짧은 기준.

## Current Baseline
- Current browser verification surface remains `apps/product-web`.
- `apps/prompt-web` remains a frozen fallback validation surface and should not receive new product features.
- This cleanup is structural preparation only. It does not implement Google login, backend persistence, user accounts, saved history, or server-owned sessions.
- The product still keeps primary results and Stage 1 follow-up / learning layers as separate visible blocks.

## Productization Cleanup Intent
Prepare `apps/product-web` so future backend API, Google login, user session, save, and history work can attach at clear boundaries.

The current cleanup boundary is:
- app/runtime structure only
- no behavior change to approval policy
- no behavior change to renderer recommendation or renderer lock behavior
- no engine-core contract changes
- no renderer output shape changes
- no silent expansion into authentication or persistence features

## Structural Bottlenecks Identified
- `ResultPanel.tsx` is still large and owns primary result display, Stage 1 follow-up UI, learning panels, copy helpers, and follow-up parsing helpers in one file.
- Product runtime HTTP calls were previously split across provider, engine, and follow-up clients.
- Provider session storage was previously embedded directly inside `useProviderSession`.

## Settled This Session
- Added `apps/product-web/src/lib/runtime/productRuntimeApiClient.ts`.
  - Centralizes product runtime HTTP calls for model listing, engine run, and Stage 1 follow-up.
  - Keeps current fallback behavior in `productEngineClient.ts` and `stage1FollowUpClient.ts`.
  - Does not add auth headers, user identity, server sessions, or persistence.
- Added `apps/product-web/src/lib/provider/providerSessionStorage.ts`.
  - Moves provider sessionStorage read/write/clear and 30-minute expiry calculation out of `useProviderSession`.
  - Keeps the current browser session credential policy unchanged.
  - API keys still live only in browser sessionStorage and still expire after 30 minutes.
- Updated `useProviderSession.ts` to call the storage boundary instead of owning raw sessionStorage access.
