## Settled
- The first screen now uses optional start templates instead of mandatory example/question/approval cards.
- Product-web renders the first result directly and leaves missing information as visible caveats and input hints.
- Plan results expose `AI 코딩툴에 넣을 내용` as a secondary handoff, not the primary product purpose.
- The coding-tool payload is generated from the current `PlanOutput` in app code, not by asking the LLM to emit JSON.
- If plan details are missing, the payload sends them through `needs_user_decision` instead of turning vague text into implementation tasks.
- Coding-agent guidance now asks the next agent to explain first, propose safe defaults, ask whether the user has their own direction, and treat short approvals like `ㅇㅇ` / `ㄱㄱ` as safe minimum scope.
- First copy attempt shows a one-time review prompt. `내용 보고 복사하기` scrolls to the result/hint area without copying; the next copy action copies directly.

## Product Boundary
This direction is acceptable only while the coding-tool payload remains a handoff harness after a plan result.

It should not become:
- the main identity of Vibe Studio
- a generic coding prompt generator
- a shortcut that encourages users to skip reading their plan result
- a path that silently expands scope for non-developer users

Keep the main product identity as:
- structured-thinking learning environment
- beginner-friendly intent clarification
- visible explanation of what is missing and why it matters

Decision question:
- Which path better protects the product identity right now: helping users build safer MVPs after plan, or helping users ask better AI questions before plan?

- Do not let AI coding handoff become the main screen or main promise.
