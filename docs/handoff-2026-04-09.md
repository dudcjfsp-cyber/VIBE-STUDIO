# Handoff

Updated: 2026-04-09 17:57 +09:00
Purpose: 다음 스레드가 오늘 구현 경계와 다음 시작점을 추측 없이 이어받도록 하기 위한 handoff

## 이번 스레드에서 실제로 끝난 범위
- `engine-contracts` 패키지 생성 완료
- `engine-core` analyze flow 구현 완료
- golden case 8개 기준 `mode_guess`, `provisional_renderer`, `next_step`, `approval_level`, `pivot_recommended` 검증 완료
- `renderer-prompt` 최소 구현 완료
- `run-engine` 최소 오케스트레이션 추가 완료
- thin app `apps/prompt-web` 추가 완료

## 현재 settled
- shared contract는 `packages/engine-contracts`를 기준으로 본다
- analyze 결과와 gate signal 계산은 `packages/engine-core/src/orchestrator/analyze-request.ts` 기준으로 본다
- app은 approval 대기를 소유해야 하므로 `run-engine`은 명시적 승인 옵션 없이는 `approval_pending` 렌더를 실행하지 않는다
- `clarify_first` 상태에서는 renderer를 실행하지 않는다
- `renderer-prompt`는 최소 내부 형식만 구현했다
- thin app은 `prompt` renderer만 연결되어 있고, 다른 renderer는 analyze 결과만 보여준다

## 현재 open
- `prompt` 결과 형식은 최소 내부 형식일 뿐 최종 제품 사양으로 고정하지 않았다
- `plan`, `architecture`, `review-report` renderer는 아직 없다
- thin app은 브라우저 수동 확인용 최소 앱이며 저장/히스토리/다중 프로젝트는 없다
- browser open 이슈는 앱 렌더 실패가 아니라 OS 창 표시/포커스 쪽 문제로 보였다

## 검증 상태
- `npm.cmd run typecheck` 통과
- `npm.cmd run verify:prompt-cycle` 통과
- 검증된 흐름:
- `direct_render` -> output 생성
- `approval_pending + required` -> 승인 없이는 output 없음
- `clarify_first` -> output 없음
- headless Chrome 기준 DOM 렌더와 JS exception 0건 확인

## 다음 스레드의 가장 자연스러운 시작점
- 우선순위 1: `renderer-plan` 또는 `renderer-architecture` 중 하나를 같은 패턴으로 추가
- 우선순위 2: thin app에서 `prompt` 외 renderer가 아직 미지원임을 더 명확히 드러내거나, renderer별 app 분기 추가
- 우선순위 3: browser open 문제를 더 다뤄야 한다면 앱 코드가 아니라 창 실행 전략/개발 서버 운영 방식부터 본다
